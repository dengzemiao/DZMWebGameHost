# DZMWebGameHost 项目架构

## 一、项目概述

Tauri 2 桌面客户端，内嵌 axum HTTP/WebSocket 服务器。用户启动服务后，同局域网设备通过浏览器访问 IP 进入游戏大厅，支持 PvP 对战、PvAI 人机、观战等模式。

**技术栈**：
- 桌面端：Tauri 2 + Vue 3 + Vite
- 后端：Rust (axum + tokio + rusqlite)
- 前端页面：原生 HTML/CSS/JS

**核心流程**：
```
Tauri 启动 → 用户点击"启动服务"
  → axum 监听 0.0.0.0:{port}
  → 玩家浏览器访问 http://{ip}:{port}
  → HTTP 提供静态页面 (game-pages/)
  → WebSocket (/ws) 实时通讯
  → SQLite 持久化玩家数据 (按 IP 记录昵称)
```

## 三、Rust 后端详解

### 3.1 AppState (state.rs)

全局应用状态，由 Tauri 管理生命周期：

```rust
pub struct AppState {
    pub server_handle: Mutex<Option<ServerHandle>>,  // 服务器句柄
    pub online_count: Arc<AtomicUsize>,              // 在线人数（与 Lobby 共享）
    pub running: Arc<AtomicBool>,                    // 服务器运行状态
    pub port: Mutex<u16>,                            // 监听端口（默认 3000）
    pub data_dir: PathBuf,                           // 数据目录（存放 game.db）
    pub game_pages_dir: PathBuf,                     // 静态页面目录
}

pub struct ServerHandle {
    pub join_handle: JoinHandle<()>,                 // tokio 任务句柄
    pub shutdown_tx: oneshot::Sender<()>,            // 关闭信号发送端
    pub broadcast_tx: broadcast::Sender<String>,     // 广播频道（用于 server_stopping）
}
```

### 3.2 Tauri 命令 (lib.rs)

| 命令 | 功能 |
|------|------|
| `start_server` | 启动 axum 服务器，返回访问地址 |
| `stop_server` | 广播 server_stopping 后关闭服务器 |
| `get_server_status` | 获取运行状态、在线人数、地址 |
| `get_local_ip` | 获取本机局域网 IP |
| `set_port` | 设置监听端口（需停止服务后） |
| `open_data_dir` | 打开数据目录 |
| `get_online_count` | 获取在线人数 |

**注**：`get_all_users` 命令已于 2026-03-19 移除（用户列表功能精简）。

**服务启动流程**：
```rust
start_server()
  → server::start(port, game_pages_dir, db_path, online_count)
    → Lobby::new(online_count, db_path)
    → http::create_router(lobby, game_pages_dir)
    → axum::Server::bind("0.0.0.0:{port}")
```

**服务停止流程**：
```rust
stop_server()
  → broadcast_tx.send("server_stopping")  // 通知所有客户端
  → sleep(200ms)                          // 等待客户端收到
  → shutdown_tx.send(())                  // 触发 graceful shutdown
  → join_handle.await                     // 等待完成
```

### 3.3 HTTP 路由 (http.rs)

```rust
Router::new()
    .route("/api/status", get(api_status))     // 返回在线人数和状态
    .route("/api/players", get(api_players))   // 返回玩家列表
```

### 3.5 Room 数据结构 (lobby.rs)

### 3.6 Lobby 核心方法

**玩家连接/断开**：
| 方法 | 说明 |
|------|------|
| `player_join(ip)` | 新连接：查找/创建 DB 玩家，自增 session_id，尝试恢复断线房间 |
| `player_leave(session_id)` | 断开：标记 RoomPlayer.disconnected=true，席位保留，清理 session_rooms |
| `restore_room_session(ip, new_session_id)` | 断线重连：通过 ip_rooms 找到原房间，更新 session_id |
| `update_player_name(session_id, name)` | 更新昵称（同步更新房间内玩家名） |

**房间管理**：
| 方法 | 说明 |
|------|------|
| `create_room(...)` | 创建房间，自动离开旧房间，PvAI 立即 Playing |
| `join_room(session_id, ip, room_id, as_spectator)` | 加入房间，支持重连识别，人满自动 Playing |
| `leave_room(session_id)` | 主动离开：清理席位，空房销毁，PvP 触发 game_over |
| `switch_role(session_id, ip, to_spectator)` | 切换观战/参战 |
| `switch_seat(session_id, target_role)` | PvP 换位（红↔黑） |
| `vote_surrender(session_id)` | 认输投票，双人游戏立即生效 |
| `restart_game(session_id)` | 再来一局：清空 game_state，广播 game_restart |

**状态管理**：
| 方法 | 说明 |
|------|------|
| `get_room_game_state(room_id)` | 获取当前局状态 |
| `record_game_state(room_id, state)` | 记录走棋后的状态 |
| `clear_room_game_state(room_id)` | 再来一局/房间销毁时清空 |

**广播方法**：
| 方法 | 说明 |
|------|------|
| `broadcast_to_room(room_id, msg)` | 广播给房间所有在线成员 |
| `broadcast_to_sessions(sessions, msg)` | 定向广播（_targets 包装） |
| `broadcast_room_update(room)` | 广播 room_updated |
| `broadcast_room_list()` | 广播 room_list_update 给所有人 |
| `broadcast_player_count()` | 广播在线人数（仅 count，不含用户列表） |

### 3.7 WebSocket 消息处理 (websocket.rs)

**连接建立流程**：
```
ws_handler() → handle_socket()
  → socket.split() → sender + receiver
  → lobby.player_join(ip) → 获取 PlayerInfo
  → 发送 welcome 消息
  → 发送 room_list 消息
  → 订阅 broadcast_rx
  → 启动 send_task（监听广播，过滤定向消息）
  → 启动 recv_task（处理客户端消息）
  → tokio::select! 等待任一任务结束
  → lobby.player_leave(session_id)
```

**定向消息机制**：
```json
// 服务端发送格式
{ "_targets": [session_id1, session_id2], "_payload": "实际消息JSON字符串" }

// send_task 过滤逻辑
if msg._targets && !msg._targets.includes(session_id):
    skip  // 不是目标，丢弃
else:
    send(msg._payload)  // 提取真实消息发送
```

| 类型 | 方向 | 字段 | 说明 |
|------|------|------|------|

| `switch_seat` | C→S | target_role | PvP 换位（如 "red"/"black"） |

### 4.5 游戏控制

| 类型 | 方向 | 字段 | 说明 |
|------|------|------|------|
| `surrender` | C→S | — | 认输 |
| `restart_game` | C→S | — | 再来一局 |
| `game_restart` | S→房间 | — | 通知所有人重置棋盘 |
| `game_over` | S→房间 | winner_role, reason | 游戏结束 |
| `game_action` | 双向 | action, ..., game_state?, from_session | 游戏动作转发 |
| `room_chat` | 双向 | content, name, system | 房间内聊天 |

**game_over reason 值**：
- `surrender` - 认输
- `opponent_leave` - 对手离开

**game_action 由服务端转发时添加 `from_session` 字段**。

### 5.1 ws-client.js (WebSocket 客户端)

**类：GameWSClient**

```js
class GameWSClient {
  // 属性
```

### 5.3 lobby.js (大厅逻辑)

**主要功能**：
- 显示连接状态、在线人数、玩家信息
- 房间列表渲染和搜索过滤
- 创建房间弹窗（游戏选择、房间类型、AI 难度）
- 修改昵称弹窗
- 大厅公共聊天

**关键流程**：
```
页面加载 → ws.connect()
  → welcome → 更新玩家信息、显示欢迎
  → room_list → 渲染房间卡片

创建房间 → 弹窗选择配置 → 存入 sessionStorage
  → 跳转 room.html?action=create
  （避免跳转时 WS 断开导致房间销毁）

加入房间 → 跳转 room.html?id={room_id}&spectator={0|1}
  （同样避免在大厅发 join_room）
```

### 5.4 room.js (房间逻辑)

**主要功能**：
- 处理创建/加入房间
- 渲染房间信息、席位、观战者
- 动态加载游戏脚本和样式
- 初始化 GameAdapter 并注入回调
- 处理游戏结束弹窗和再来一局
- 处理身份切换（观战↔参战）
- 房间内聊天

**URL 参数**：
- `?action=create` - 创建房间（从 sessionStorage 读取配置）
- `?id={room_id}` - 加入房间
- `&spectator=1` - 以观战身份加入

**关键状态**：
```js
let roomData = null;      // 当前房间完整数据
let currentRoomId = null; // 房间 ID
let gameAdapter = null;   // 游戏适配器实例
let isSpectator = false;  // 是否观战
let isPlayer = false;     // 是否参战
let hasJoined = false;    // 避免重复加入
```

**核心流程**：

1. **入房流程**：
```
ws.connect() → welcome → enterRoom()
  ├── 有 room_id → join_room
  └── action=create → create_room（配置从 sessionStorage 读取）

room_created/room_joined → applyRoomData() + maybeLoadGame()
  → 动态注入 game.js/game.css
  → script.onload → initGameAdapter()
    → new GameAdapter(container, config)
    → 注入 sendAction/notifyGameOver 回调
    → adapter.init()
    → 若有 game_state → adapter.restoreGameState()
    → 若 state=playing → adapter.onGameStart()
```

2. **room_updated 处理**：
```
room_updated → applyRoomData()
  → 更新 UI（席位、观战者、按钮）
  → 检测状态变化：
    ├── waiting → playing：调用 onGameStart()
    ├── playing → waiting（玩家减少）：调用 onOpponentLeave()
    └── 身份变化（参战↔观战）：更新 adapter.isSpectator/role + 重新渲染
```

3. **game_restart 处理**：
```
game_restart → 隐藏结算弹窗
  → adapter.reset()
  → adapter.gameStarted = true
  → adapter.onGameStart()
```

4. **游戏结束处理**：
```
game_over / adapter.notifyGameOver()
  → handleGameOver(data)
    ├── 观战者：仅聊天区提示
    └── 参战者：显示结算弹窗（胜/负/认输）
```

## 六、游戏插件化架构

### 6.1 GameAdapter 标准接口

每款游戏需实现以下接口：

```js
class XxxGameAdapter {
  // 构造函数
  constructor(container, config)
  // config: { role, roomType, aiDifficulty, mySessionId, isSpectator }

  // 必须实现
  init()                        // 初始化 DOM 和状态
  onGameStart(roomData)         // 游戏开始（设置 gameStarted=true）
  onRemoteAction(data)          // 收到远程动作
  onOpponentLeave()             // 对手离开
  reset(reason?)                // 重置棋盘

  // 状态同步（推荐实现）
  restoreGameState(state)       // 从服务端状态恢复
  getStateSnapshot()            // 获取当前局状态快照

  // 由 room.js 注入
  sendAction(data)              // 发送 game_action
  notifyGameOver(result)        // 通知游戏结束
}
```

### 6.2 新增游戏步骤

1. 创建 `game-pages/games/xxx/` 目录
2. 实现 `game.js`（导出 `XxxGameAdapter` 到 window）
3. 实现 `game.css`

无需修改 lobby.js、room.js、后端代码。

## 七、断线重连机制

### 7.1 服务端处理

```
WS 断开 → player_leave(session_id)
  → 找到该 session 的 IP
  → 在房间中标记 RoomPlayer.disconnected = true
  → 保留 ip_rooms 映射（不清除）
  → 清除 session_rooms 映射
  → 广播 room_updated（其他人看到该玩家离线）

同 IP 重连 → player_join(ip)
  → 检查 ip_rooms 是否有记录
  → restore_room_session(ip, new_session_id)
    → 更新 RoomPlayer.session_id
    → 设置 disconnected = false
    → 发送 room_joined(reconnected: true, game_state)
    → 广播 room_updated
```

### 7.2 客户端处理

```
ws.on('room_joined', (room) => {
  if (room.reconnected) {
    // 断线重连场景
  }
  applyRoomData(room);
  maybeLoadGame(room);  // 游戏脚本可能已加载
});

// 若有 game_state，在 initGameAdapter 中调用：
adapter.restoreGameState(room.game_state);
```

## 九、常见问题定位

### 9.1 房间相关

| 问题 | 可能原因 | 排查点 |
|------|---------|--------|
| 房间创建后立即消失 | 页面跳转时 WS 断开 | 检查 sessionStorage 配置、room.js 入房逻辑 |
| 断线后无法重连房间 | ip_rooms 映射丢失 | 检查 player_leave 是否清理了 ip_rooms |
| 观战者看不到棋盘更新 | game_action 未转发 | 检查 get_all_session_ids 是否包含观战者 |
| 换位后角色错误 | switch_seat 逻辑 | 检查 assign_role 和前端 role 更新 |

### 9.2 游戏相关

| 问题 | 可能原因 | 排查点 |
|------|---------|--------|
| 游戏脚本加载失败 | 路径错误或已加载 | 检查 game-registry.js scriptPath |
| onGameStart 不触发 | 状态判断错误 | 检查 room.state 和 room_type |
| 观战者能操作 | isSpectator 未设置 | 检查 config.isSpectator 和 _canControl |
| 走棋不同步 | game_action 未发送 | 检查 sendAction 调用和 game_state |

### 9.3 WebSocket 相关

| 问题 | 可能原因 | 排查点 |
|------|---------|--------|
| 消息丢失 | 定向消息过滤 | 检查 _targets 是否包含目标 session |
| 重连不工作 | serverStopped=true | 检查 server_stopping 是否被触发 |
| 广播不生效 | broadcast_tx 错误 | 检查 Lobby 广播方法 |
